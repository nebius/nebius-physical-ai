# MK8s CLI Audit: nebius mk8s vs npa cluster

**Date:** 20260515T002328Z
**Author:** Codex (W8-mk8s-audit run)
**Status:** Draft for operator review

## Summary

`npa cluster` should not become a second `nebius mk8s` CLI. The local
Nebius CLI already provides the raw managed-Kubernetes lifecycle surface:
cluster create/delete/list/get, node-group create/delete/list/get, kubeconfig
generation, operation waiting, Kubernetes version listing, and node-group
compatibility discovery. `npa cluster` adds real but narrower value as an
NPA-managed execution-target profile: local state, named kubeconfig caching,
NPA project/subnet defaults, readiness polling across cluster and node groups,
and GPU alias/default handling. The right move is Option B: shrink the intended
scope of `npa cluster` to NPA-specific value and defer raw MK8s administration
to `nebius mk8s`.

## What nebius mk8s already provides

The installed `nebius mk8s` surface discovered in this run has 32 commands
under three top-level groups: `cluster`, `node-group`, and `v1alpha1`.

Cluster lifecycle:

- `nebius mk8s cluster create --parent-id <project> --control-plane-subnet-id <subnet> --name <name>`
- `nebius mk8s cluster delete --id <cluster-id>`
- `nebius mk8s cluster list --parent-id <project>`
- `nebius mk8s cluster get --id <cluster-id>`
- `nebius mk8s cluster get-by-name --parent-id <project> --name <name>`

Cluster credentials and versions:

- `nebius mk8s cluster get-credentials --id <cluster-id> --kubeconfig <path> --context-name <name> --external`
- `nebius mk8s cluster get-token`
- `nebius mk8s cluster list-control-plane-versions`

Node-group lifecycle:

- `nebius mk8s node-group create --parent-id <cluster-id> --template-resources-platform <platform> --template-resources-preset <preset>`
- `nebius mk8s node-group delete --id <node-group-id>`
- `nebius mk8s node-group list --parent-id <cluster-id>`
- `nebius mk8s node-group get --id <node-group-id>`
- `nebius mk8s node-group get-by-name --parent-id <cluster-id> --name <name>`
- `nebius mk8s node-group upgrade --id <node-group-id> --latest-infra-version`

Operational and discovery commands:

- `nebius mk8s cluster operation get|list|wait`
- `nebius mk8s node-group operation get|list|wait`
- `nebius mk8s node-group get-compatibility-matrix --cluster-kubernetes-version <version>`

The Nebius CLI also has global `--format yaml|json|jsonpath|table|text`,
`--file` request-body support, JSON/YAML positional request schemas, and
`--async` on mutating operations. Help text says mutating operations wait for
operation completion by default unless `--async` is set.

## What npa cluster adds today

`npa cluster` is a subprocess wrapper around `nebius mk8s`, not an independent
Kubernetes API client. Its useful additions are:

- Local cluster workspace at `~/.npa/clusters/<name>/`.
- Local `cluster.json`, `metadata.json`, node-group cache, and kubeconfig path.
- Kubeconfig convention: `~/.npa/clusters/<name>/kubeconfig` with context name
  equal to the NPA cluster name.
- Project resolution from `--project-id`, NPA environment variables, or
  `~/.npa/config.yaml`.
- Subnet resolution through the shared NPA subnet resolver.
- Opinionated bootstrap defaults: `eu-north1`, Kubernetes `1.33`, `cpu-e2`,
  `2vcpu-8gb`, one CPU node, public control-plane endpoint, and etcd size `1`.
- Readiness polling that checks cluster status plus node-group status and node
  count, rather than only operation completion.
- GPU aliases and defaults for `h100`, `h200`, `l40s`, and `rtx6000`.
- Simpler node-group options for `--public-ip`, `--autoscaling-min`, and
  `--autoscaling-max`.

These are Workbench target-management conveniences. They are not a reason to
mirror every `nebius mk8s` command.

## What npa cluster duplicates from nebius mk8s

Every current `npa cluster` command sits on top of an existing Nebius lifecycle
primitive:

- `npa cluster deploy` wraps `cluster create`, `node-group create`, and
  `cluster get-credentials`.
- `npa cluster destroy` wraps `cluster delete`.
- `npa cluster list` wraps `cluster list`.
- `npa cluster status` wraps `cluster get`, `cluster get-by-name`, and
  `node-group list`.
- `npa cluster node-group add` wraps `node-group create`.
- `npa cluster node-group remove` wraps `node-group delete`.
- `npa cluster node-group list` wraps `node-group list`.
- `npa cluster node-group status` wraps `node-group get`, `get-by-name`, and
  `list`.

The command-level duplication is acceptable only where the NPA layer changes
the product contract: state, defaults, kubeconfig convention, polling, GPU
aliases, and workflow target selection.

## Recommendation Options

### Option A: Keep `npa cluster` as-is

This is acceptable only as a short-term milestone artifact. It preserves the
existing UX and tests, but it risks teaching users that NPA is the preferred
surface for all MK8s administration. That is the wrong product boundary because
Nebius already owns and evolves the canonical MK8s CLI.

### Option B: Shrink `npa cluster` to focus on NPA-genuine value

This is the recommended option. Keep `npa cluster` as a narrow Workbench
execution-target helper:

- Bootstrap an NPA-managed cluster with known-good defaults.
- Cache and name kubeconfig/state for later workflow commands.
- Attach simple GPU pools using NPA aliases and defaults.
- Show NPA target status by merging local state and remote state.
- Clean up NPA-managed local state when destroying resources.

Do not expand `npa cluster` toward raw MK8s parity. Advanced lifecycle,
upgrade, edit, update, operation inspection, version discovery, and compatibility
matrix workflows should be documented as direct `nebius mk8s` commands. If NPA
needs compatibility awareness, add validation that consumes Nebius discovery
output; do not add a parallel discovery CLI unless a workflow command needs it.

### Option C: Defer all cluster lifecycle to `nebius mk8s`

This is cleaner at the cloud-control-plane boundary but too aggressive today.
It would remove the only current NPA convention for local kubeconfig placement,
named target state, and future workflow binding. Workbench still needs a stable
way to say "run this workflow on the NPA cluster named X" without requiring the
operator to hand-manage kubeconfig paths and local metadata.

## Recommended Option

Choose Option B.

`npa cluster` was the right investment only if treated as an NPA target/profile
layer. It is the wrong investment if treated as a replacement for `nebius mk8s`.
The audit found one pure duplicate capability, kubeconfig retrieval, and many
value-add wrappers whose value is the NPA convention around the Nebius call, not
the lifecycle call itself. The scope line should be explicit in docs and future
milestones:

> Use `nebius mk8s` for general Managed Kubernetes administration. Use
> `npa cluster` only to create, select, inspect, and clean up NPA-managed
> Workbench execution targets.

## Migration Path

No command deletion is required immediately. The safer migration is to constrain
scope before adding more surface area:

1. Documentation commit: define `npa cluster` as an NPA-managed target/profile
   helper and link raw administration to `nebius mk8s`.
2. CLI help wording commit: adjust help text away from generic "Manage Nebius
   Managed Kubernetes clusters" toward "Manage NPA Workbench cluster targets."
3. Validation commit: before future GPU-pool expansion, check Nebius
   compatibility data for Kubernetes version, platform, OS, and driver preset
   drift instead of hardcoding more parallel knowledge.
4. Workflow integration commit: make future `npa workflow` commands consume
   `~/.npa/clusters/<name>/kubeconfig` and state as the target registry.

Estimated migration cost: 2 to 4 commits, low implementation risk, moderate UX
risk if users have already started treating `npa cluster` as a general MK8s CLI.

## Risk Surfaces

- Keeping the current command names may still imply NPA is a full MK8s
  lifecycle layer unless help/docs are sharpened.
- Hardcoded NPA GPU/platform defaults can drift from Nebius compatibility
  output. This should be validated against `node-group get-compatibility-matrix`
  before more GPU presets are added.
- Removing `npa cluster` entirely would break future `npa workflow` integration
  plans around named kubeconfig/state.
- Wholly relying on `nebius mk8s` ties NPA execution-target bootstrap to Nebius
  CLI behavior and versioning, but wrapping more deeply would increase that
  coupling, not reduce it.

## Out of Scope

- Workflow engine selection.
- Argo installation.
- Tool-as-pod packaging.
- Workflow CLI design beyond the cluster-target boundary.
- Real cluster provisioning or Nebius Job submission.
