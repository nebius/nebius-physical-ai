# Reproducing the 8-GPU H200 Physical AI Demo

This runbook recreates the Cosmos, Isaac Lab, GR00T, and FiftyOne demo flow in
your own Nebius project. It uses placeholders only; replace every value in
angle brackets before running commands.

## Preconditions

- A Nebius tenant and project with quota for one 8-GPU H200 VM and one L40S VM.
- `npa` installed from this repository and available on `PATH`.
- `~/.npa/credentials.yaml` configured with the Nebius S3 credentials, HF token,
  and any NGC token needed by your GR00T path.
- A target S3 bucket owned by your project.
- Scoped S3 grants that allow your active principal to read the source artifact
  manifest and write to your target bucket.
- SSH access to the BYOVM hosts, or use managed `deploy` commands instead of
  BYOVM registration.
- Public ingress permission for ports used by the demo services.
- Terraform and the Nebius CLI configured if you provision new managed VMs.

The default demo artifact manifest reads from a Nebius-owned source bucket. If
your principal cannot read that source, ask for read access or provide a custom
manifest with `npa demo stage --manifest`.

```bash
export REPO_PATH="<REPO_PATH>"

export PROJECT_ID="<YOUR_PROJECT_ID>"
export TENANT_ID="<YOUR_TENANT_ID>"
export REGION="<YOUR_REGION>"

export TARGET_BUCKET="<YOUR_BUCKET>"
export TARGET_BUCKET_URI="s3://$TARGET_BUCKET/demo-8gpu-h200"

export H200_VM_IP="<H200_VM_IP>"
export H200_INSTANCE_ID="<H200_INSTANCE_ID>"
export ISAAC_VM_IP="<L40S_VM_IP>"
export ISAAC_INSTANCE_ID="<L40S_INSTANCE_ID>"
export SSH_USER="ubuntu"

export SOURCE_CIDR="<YOUR_SOURCE_CIDR>"

export COSMOS_ALIAS="<COSMOS_ALIAS>"
export GROOT_ALIAS="<GROOT_ALIAS>"

export FIFTYONE_PORT="5151"

export H200_GPU_PRESET="<H200_GPU_PRESET>"
export L40S_GPU_COUNT="1"
export L40S_GPU_TYPE="<L40S_GPU_TYPE>"
export L40S_GPU_PRESET="<L40S_GPU_PRESET>"

export COSMOS_MODEL="nvidia/Cosmos-1.0-Diffusion-7B-Text2World"
export GROOT_MODEL="nvidia/GR00T-N1.7-3B"
```

## Step 1: Stage Demo Artifacts

Stage all portable demo artifacts into your bucket, then verify the staged copy
using S3 HEAD/list operations.

```bash
npa demo stage --target-bucket "$TARGET_BUCKET_URI"
npa demo verify --target-bucket "$TARGET_BUCKET_URI"
```

The stage command is idempotent. File artifacts are skipped when
`x-amz-meta-sha256` matches the manifest; prefix artifacts are checked by object
count and total size.

## Step 2: Provision or Register VMs

If you already created BYOVM hosts, register them as workbench aliases:

```bash
npa workbench cosmos -p "$PROJECT_ALIAS" register-byovm \
  --alias "$COSMOS_ALIAS" \
  --instance-id "$H200_INSTANCE_ID" \
  --port "$COSMOS_PORT"

npa workbench groot -p "$PROJECT_ALIAS" register-byovm \
  --alias "$GROOT_ALIAS" \
  --instance-id "$H200_INSTANCE_ID" \
  --port "$GROOT_PORT"

npa workbench fiftyone -p "$PROJECT_ALIAS" register-byovm \
  --alias "$FIFTYONE_ALIAS" \
  --instance-id "$H200_INSTANCE_ID" \
  --port "$FIFTYONE_PORT"
```

If you do not have BYOVM hosts, use the corresponding `deploy` commands with
`--project-id`, `--tenant-id`, `--region`, `--gpu-type`, and `--gpu-preset`.

## Step 3: Ensure Network Ingress

Open the demo service ports to your source CIDR.

```bash
npa network ensure-ingress \
  --vm "$H200_INSTANCE_ID" \
  --ports "$FIFTYONE_PORT,$COSMOS_PORT,$GROOT_PORT" \
  --source "$SOURCE_CIDR" \
  --tool physical-ai-demo
```

## Step 4: Deploy the Tool Services on BYOVM

Deploy Cosmos, GR00T, and FiftyOne onto the H200 host.

```bash
npa workbench cosmos -p "$PROJECT_ALIAS" -n "$COSMOS_ALIAS" deploy \
  --runtime byovm \
  --host "$H200_VM_IP" \
  --ssh-user "$SSH_USER" \
  --ssh-key "$SSH_KEY" \
  --project-id "$PROJECT_ID" \
  --tenant-id "$TENANT_ID" \
  --region "$REGION" \
  --gpu-type "$H200_GPU_TYPE" \
  --gpu-preset "$H200_GPU_PRESET" \
  --gpu-count "$H200_GPU_COUNT" \
  --server-port "$COSMOS_PORT" \
  --model "$COSMOS_MODEL"

npa workbench groot -p "$PROJECT_ALIAS" -n "$GROOT_ALIAS" deploy \
  --runtime byovm \
  --host "$H200_VM_IP" \
  --ssh-user "$SSH_USER" \
  --ssh-key "$SSH_KEY" \
  --project-id "$PROJECT_ID" \
  --tenant-id "$TENANT_ID" \
  --region "$REGION" \
  --gpu-type "$H200_GPU_TYPE" \
  --gpu-preset "$H200_GPU_PRESET" \
  --gpu-count "$H200_GPU_COUNT" \
  --server-port "$GROOT_PORT" \
  --model "$GROOT_MODEL"

npa workbench fiftyone -p "$PROJECT_ALIAS" -n "$FIFTYONE_ALIAS" deploy \
  --runtime byovm \
  --host "$H200_VM_IP" \
  --ssh-user "$SSH_USER" \
  --ssh-key "$SSH_KEY" \
  --project-id "$PROJECT_ID" \
  --tenant-id "$TENANT_ID" \
  --region "$REGION" \
  --gpu-count "$H200_GPU_COUNT"
```

Deploy or register Isaac Lab on the L40S host:

```bash
npa workbench isaac-lab -p "$PROJECT_ALIAS" -n "$ISAAC_ALIAS" deploy \
  --runtime byovm \
  --host "$ISAAC_VM_IP" \
  --ssh-user "$SSH_USER" \
  --ssh-key "$SSH_KEY" \
  --project-id "$PROJECT_ID" \
  --tenant-id "$TENANT_ID" \
  --region "$REGION" \
  --gpu-type "$L40S_GPU_TYPE" \
  --gpu-preset "$L40S_GPU_PRESET" \
  --gpu-count "$L40S_GPU_COUNT"
```

## Step 5: Isaac Lab Training

Run a short Isaac Lab training job when you want a fresh checkpoint. For the
recorded demo path, the staged LeRobot export is enough and this step can be
skipped.

```bash
npa workbench isaac-lab -p "$PROJECT_ALIAS" -n "$ISAAC_ALIAS" train \
  --task "$ISAAC_TASK" \
  --num-envs 64 \
  --steps 1000 \
  --output-path "$TARGET_BUCKET_URI/isaac-lab-train/$ISAAC_TASK/" \
  --output-format json
```

## Step 6: Convert to LeRobot and GR00T Layouts

Export Isaac Lab rollouts to the standard LeRobot layout:

```bash
npa workbench isaac-lab -p "$PROJECT_ALIAS" -n "$ISAAC_ALIAS" export-lerobot \
  --task "$ISAAC_TASK" \
  --num-episodes 10 \
  --steps-per-episode 30 \
  --output-path "$ISAAC_EXPORT" \
  --output-format json
```

Convert the standard LeRobot dataset to the GR00T LeRobot layout:

```bash
npa workbench groot -p "$PROJECT_ALIAS" -n "$GROOT_ALIAS" convert \
  --input-path "$ISAAC_EXPORT" \
  --output-path "$GROOT_DATASET" \
  --direction lerobot-to-groot \
  --robot-embodiment "$GROOT_EMBODIMENT" \
  --output-format json
```

## Step 7: GR00T Inference

Run GR00T inference on the staged or freshly converted dataset. Use
`--allow-host-creds` only when you intentionally want to fall back from scoped
local S3 credentials to the VM service credentials.

```bash
npa workbench groot -p "$PROJECT_ALIAS" -n "$GROOT_ALIAS" infer \
  --input-path "$GROOT_MODEL" \
  --dataset-path "$GROOT_DATASET" \
  --output-path "$GROOT_OUTPUT" \
  --embodiment-tag "$GROOT_EMBODIMENT" \
  --steps 32 \
  --output json
```

## Step 8: Visualization

Create local Rerun recordings from the staged LeRobot dataset and predictions:

```bash
mkdir -p "$LOCAL_ARTIFACT_DIR"

npa convert lerobot-to-rrd \
  --input-path "$GROOT_DATASET" \
  --duration 5 \
  --output-path "$ISAAC_RRD"

npa convert lerobot-to-rrd \
  --input-path "$GROOT_DATASET" \
  --predictions-path "$GROOT_PREDS" \
  --duration 5 \
  --output-path "$GROOT_RRD"
```

For MP4 playback instead of Rerun recordings:

```bash
npa viz lerobot \
  --input-path "$GROOT_DATASET" \
  --backend matplotlib \
  --duration 5 \
  --output-path "$LOCAL_ARTIFACT_DIR/isaac-lab-trajectory.mp4" \
  --title "Isaac Lab G1 trajectory"

npa viz lerobot \
  --input-path "$GROOT_DATASET" \
  --predictions-path "$GROOT_PREDS" \
  --backend matplotlib \
  --layout overlay \
  --duration 5 \
  --output-path "$LOCAL_ARTIFACT_DIR/groot-predictions-overlay.mp4" \
  --title "Isaac Lab input vs GR00T predictions"
```

## Step 9: FiftyOne Curation

Load the staged Cosmos samples into FiftyOne:

```bash
npa workbench fiftyone -p "$PROJECT_ALIAS" -n "$FIFTYONE_ALIAS" load-dataset \
  --name "$FIFTYONE_DATASET" \
  --input-path "$FIFTYONE_SOURCE" \
  --format auto

npa workbench fiftyone -p "$PROJECT_ALIAS" -n "$FIFTYONE_ALIAS" datasets list \
  --search "$FIFTYONE_DATASET"
```

Open the app:

```bash
npa workbench fiftyone -p "$PROJECT_ALIAS" -n "$FIFTYONE_ALIAS" launch
```

## Step 10: Teardown Checklist

The current CLI does not provide a one-command demo teardown. Track resources
manually and remove them in this order:

```bash
npa workbench fiftyone -p "$PROJECT_ALIAS" -n "$FIFTYONE_ALIAS" deploy --destroy
npa workbench groot -p "$PROJECT_ALIAS" -n "$GROOT_ALIAS" deploy --destroy
npa workbench cosmos -p "$PROJECT_ALIAS" -n "$COSMOS_ALIAS" deploy --destroy
npa workbench isaac-lab -p "$PROJECT_ALIAS" -n "$ISAAC_ALIAS" deploy --destroy
```

For BYOVM hosts, also stop or delete the underlying compute instances through
your normal Nebius project workflow.

## Troubleshooting

Auth failures:
Run `npa demo verify --target-bucket "$TARGET_BUCKET_URI"` first. If it fails,
fix S3 grants before deploying services.

Ingress failures:
Re-run `npa network ensure-ingress` with the VM instance ID, required ports, and
your current source CIDR.

GPU not found:
Check the VM shape and the `--gpu-count`, `--gpu-type`, and `--gpu-preset`
values. BYOVM deploys detect GPUs over SSH.

Artifact missing:
Re-run `npa demo stage --target-bucket "$TARGET_BUCKET_URI"`. Objects with
matching sha256 metadata are skipped.

GR00T upload fallback:
If scoped local credentials cannot write to the target bucket and you
intentionally want to use the VM service credentials, pass `--allow-host-creds`
on commands that support it. Do not use the flag to mask unknown S3 failures.
